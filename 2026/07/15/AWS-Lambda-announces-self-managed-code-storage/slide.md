# AWS Lambda、自己管理型コード保存機能を発表

Amazon S3からの直接参照とコード保存制限の廃止

**発表日: 2026年7月15日**

---

## 概要

### 3つの主要な改善

- **コード保存制限の廃止**: S3バケットの容量まで関数とレイヤーコードを保存可能
- **アクティベーション時間の短縮**: 中間コピー不要により関数作成・更新が高速化
- **デフォルト制限の拡大**: Lambda管理型ストレージが75GBから300GBに増量

### 自己管理型ストレージの特徴

- Amazon S3バケットから直接ソースコードを参照
- 追加のLambda料金不要（S3ストレージ料金のみ）
- すべての商用AWSリージョンで利用可能

# 前提・背景

従来の制限とその課題

---

## 従来の課題

### Lambda管理ストレージの制限

- **リージョンあたり75GB**: 大規模なコード保存には不足
- **クォータ増加の手続き**: サポートチケット提出が必要
- **中間コピーの処理**: 無駄な時間とリソース消費

### 顧客の悩み

- 多数の関数やレイヤーをデプロイする場合、容量制限に抵触
- 関数更新のたびにLambda管理ストレージへのコピーが発生
- デプロイメントパッケージの一元管理ができない

---

## 市場背景

### サーバーレス時代の進化

- マイクロサービスアーキテクチャの広がり
- 関数の数と複雑さの増加
- S3との連携強化（S3 Files、S3 Annotationsなど）

### AWS の新戦略

- S3の機能拡張（高性能NFS、メタデータ管理）
- Lambdaとストレージの統合深化
- 顧客のコスト最適化と運用効率向上

# 変更内容・新機能

自己管理型コード保存の詳細

---

## 自己管理型コード保存の仕組み

### 従来の流れ（廃止）

```
S3バケット → Lambda管理ストレージにコピー → 関数実行
```

### 新しい流れ（推奨）

```
S3バケット → 直接参照 → 関数実行
```

### メリット

- コピー処理が不要 → **アクティベーション時間が短縮**
- 中間ストレージ不要 → **コスト削減**
- S3容量まで保存可能 → **制限なし**

---

## 設定方法

### パラメータ設定

関数またはレイヤーを作成・更新する際に以下を実施：

```json
{
  "S3ObjectStorageMode": "REFERENCE"
}
```

### 利用可能なツール

- **AWS CLI**: コマンドラインから設定
- **AWS CloudFormation**: Infrastructure as Code
- **AWS SAM**: サーバーレスアプリケーション定義
- **AWS SDKs**: プログラマティック設定
- **Lambda Console**: GUIから設定

---

## IAM権限の設定

### 必要な権限

Lambda service principalに対して以下の権限を付与：

| 権限 | 説明 |
|------|------|
| `s3:GetObject` | オブジェクトの読み取り |
| `s3:GetObjectVersion` | バージョン履歴の読み取り |

### 権限付与の例

```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "lambda.amazonaws.com"
  },
  "Action": [
    "s3:GetObject",
    "s3:GetObjectVersion"
  ],
  "Resource": "arn:aws:s3:::your-bucket/*"
}
```

# 効果・メリット

利点と改善点

---

## 主なメリット

### パフォーマンス向上

- **関数アクティベーション時間が短縮**: 中間コピー不要
- **更新速度の向上**: 大規模なコードでも迅速に対応
- **レイテンシの削減**: S3からの直接参照

### スケーラビリティ向上

- **コード保存制限の廃止**: S3バケット容量まで対応
- **複数関数の管理**: 大規模アプリケーション対応
- **レイヤー管理**: サイズ制限なしで共有コード管理

### コスト最適化

- **追加Lambda料金なし**: S3ストレージ料金のみ
- **デプロイ効率化**: 無駄な処理削減
- **運用コスト削減**: クォータ増加申請が不要

---

## Lambda管理型ストレージの拡張

### デフォルト制限の増加

| 項目 | 従来 | 現在 | 増加率 |
|------|------|------|--------|
| **リージョンあたりの制限** | 75 GB | 300 GB | **4倍** |

### 自己管理型との使い分け

- **小～中規模**: Lambda管理型（75GB → 300GB）
- **大規模**: 自己管理型S3（容量無制限）

---

## デプロイメント管理の一元化

### 利点

- **単一の情報源**: 顧客アカウント内でパッケージ管理
- **バージョン管理**: S3 Versioning で履歴保持
- **CI/CDの簡素化**: S3 Update → 自動反映
- **ポータビリティ**: 他環境への移行が容易

# ユースケース

実際の活用例

---

## ユースケース1: 大規模マイクロサービス

### 課題

- 100個以上の関数をデプロイ
- 各関数が10MB～数100MBのコード
- 従来：75GBを超えるため、クォータ申請が必要

### 解決策

- 自己管理型S3ストレージを採用
- S3に全関数コードを一元管理
- 制限なくスケーラブルに対応

---

## ユースケース2: データ処理パイプライン

### 課題

- 複数のデータ処理関数
- 共有ライブラリを複数のレイヤーで使用
- 更新のたびに中間コピーが発生 → 遅延

### 解決策

- 自己管理型S3 + S3 Versioning
- 更新 → 即座にLambdaが新バージョンを参照
- デプロイ時間が大幅短縮

---

## ユースケース3: 機械学習モデル配置

### 課題

- 大規模MLモデル（数GB）を関数コードに含める
- 従来は保存制限でサポートチケット必須

### 解決策

- S3にMLモデルを保存
- Lambda が直接S3から参照
- 複数の関数で同じモデルを共有可能
