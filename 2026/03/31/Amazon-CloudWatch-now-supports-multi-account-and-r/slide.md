# Amazon CloudWatch
データソース基準でのログ一元化

**発表日: 2026年3月30日**

---

## 概要

### CloudWatchログ一元化機能の進化

- **データソース名とタイプ**に基づくルール定義が可能に
- 複数のAWSアカウント・リージョンのログを**単一の宛先アカウント**に集約
- ロググループ名ではなく、**データソース種別**で一元化ルールを指定可能
- CloudTrail、VPC Flow Logs、EKS Audit Logsなど、複数のサービスログに対応

---

## 前提・背景

### マルチアカウント環境のログ管理の課題

- **複数アカウント**: 組織全体のログが分散して管理が困難
- **複数リージョン**: 拠点別・リージョン別に異なるロググループが存在
- **属性管理の負担**: 個別のロググループ名を把握・管理する手間
- **セキュリティ監視**: 組織全体の統一的なセキュリティ監視が難しい

### CloudWatchの進化

- 2026年2月: カスタム宛先ロググループ名に対応
- 2026年3月（今回）: **データソース種別による一元化**で管理を簡素化

---

## 変更内容・新機能

### 1. データソース基準の一元化ルール定義

**従来の方法**
- 個別のロググループ名を指定して一元化
- 新しいロググループが増えるたびにルール更新が必要

**新しい方法**
- `VPC Flow Logs`、`CloudTrail Logs`、`EKS Audit Logs` などで直接指定
- ロググループ名を知らなくても、データソース種別で一元化可能

---

## 変更内容・新機能（続き）

### 2. データソース検出方式

#### AWSサービスログ
- CloudWatch が**自動的に検出**
- ロググループのメタデータから種別を判定

#### アプリケーションログ
- **ロググループタグ**に基づいて識別
- タグで任意のデータソース分類が可能

### 3. 利用可能な対象データソース

- **CloudTrail Logs** - API監査ログ
- **VPC Flow Logs** - ネットワークトラフィックログ
- **EKS Audit Logs** - Kubernetes監査ログ
- その他AWSサービスログ
- **カスタムタグベースのアプリケーションログ**

---

## 設定方法

### ステップ1: 一元化ルールの作成・変更

**CloudWatchコンソール**
- CloudWatch Logs → ログの一元化 → ルール作成

**AWS CLI / AWS SDK**
- `create-log-centralization-rule` コマンド

### ステップ2: データソース選択基準を指定

```
[ルール設定画面]
- データソース名: CloudTrail
- データソース種別: Logs
- 対象リージョン: すべて（またはカスタム）
- 宛先アカウント: 中央アカウントID
```

---

## 効果・メリット

### 1. 運用効率化

- **ロググループ名管理の廃止**: データソース名で統一
- **自動検出**: 新規ロググループが自動的に一元化対象に
- **スケーラビリティ**: 組織成長時の追加作業が最小限

### 2. セキュリティ強化

- **中央監視の実現**: 全社統一のセキュリティ分析が可能
- **CloudTrail統一管理**: 全APIアクティビティを一箇所で監視
- **VPC Flow ログ統一**: ネットワーク異常検知が容易

### 3. コスト効率

- **標準CloudWatch Logs料金**: ログの取り込み・保存・転送に標準価格適用
- **重複排除**: 同じログの複数コピーを回避

---

## 効果・メリット（続き）

### 4. ガバナンス向上

- **タグベース管理**: アプリケーション分類が柔軟に
- **AWS Organizations連携**: 組織全体ポリシーの一元適用
- **監査対応**: ログ一元化ルールの透明性向上

## 適用リージョンと料金

### 利用可能リージョン

- **すべてのAWSコマーシャルリージョン**（CloudWatchログ一元化がサポートされている地域）

### 料金体系

| 費用項目 | 料金 |
|---------|------|
| ログの取り込み | 標準CloudWatch Logs料金 |
| ストレージ | 標準CloudWatch Logs料金 |
| クロスアカウント/リージョン転送 | 標準CloudWatch Logs料金 |

## ベストプラクティス

### 1. タグ戦略の策定

```
アプリケーションログに対して
- Environment: Production / Staging / Development
- Application: WebServer / Database / Cache
- DataSource: CustomApp (タグベース)
```

### 2. ルール設計

- **最小権限**: 必要なデータソースのみを選択
- **定期レビュー**: 新規ロググループの自動一元化対象確認

### 3. 監視とアラート

- **中央アカウント**: ログ配信失敗時の監視
- **KMS権限**: 暗号化ログの場合、キー権限を確認

## まとめ

### Amazon CloudWatchログ一元化（データソース基準）の価値

1. **シンプルな管理**: ロググループ名から解放
2. **スケーラブル**: 新規リソース自動対応
3. **セキュリティ強化**: 中央監視を容易に実装
4. **自動検出**: AWSサービスログは自動判定
5. **組織規模での運用**: マルチアカウント組織に最適

### 次のステップ

1. 現在のログ一元化ルールを棚卸し
2. データソース選択基準に変換可能か確認
3. パイロット環境でテスト
4. 段階的に本番環境へ展開
