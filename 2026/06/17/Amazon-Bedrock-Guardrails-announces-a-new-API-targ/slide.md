# Amazon Bedrock Guardrailsがエージェンティック AI ワークフロー向け新APIを発表

新しいInvokeGuardrailChecks API でリソースレスなセーフガード実装が可能に

**発表日: 2026年6月16日**

---

## 概要

### InvokeGuardrailChecks API の主な特徴

- **リソースレス設計**: ガードレール リソースを作成することなく個別のセーフガードを適用可能
- **ステップごとの柔軟性**: エージェンティック AI ループの任意のポイントで異なるセーフガード構成を実装
- **グラニュラ制御**: リクエストごとにどのセーフガードを実行するかを指定し、カスタムしきい値とアクションを設定

### 対応するセーフガード

- **コンテンツフィルタ**: ヘイト、暴力、性的、侮辱、不正行為などの有害コンテンツを検出
- **プロンプト攻撃検出**: ジェイルブレイク、プロンプトインジェクション、プロンプトリークを独立したチェックとして識別
- **機密情報フィルタ**: サポートされている PII エンティティタイプを検出

---

## 前提・背景

### エージェンティック AI の課題

- **複数ステップの処理**: エージェンティック AI アプリケーションは計画、ツール呼び出し、出力処理、反復などの複数ステップを実行
- **ステップごとのリスク差異**: 各ステップは異なるリスク プロファイルを持ち、一律のガードレールでは対応困難
- **ワークフロー の進化**: 開発時にセーフガードを柔軟に追加、削除、調整する必要性

### 従来のアプローチの限界

- ガードレール ID の追跡とバージョン管理が必要
- ガードレール リソースの作成・管理に手間がかかる
- ステップごとの異なるリスク プロファイルへの対応が複雑

---

## 主な機能

### 1. リソースレス API 設計

- ガードレール リソースの作成が不要
- 各リクエストでセーフガード構成を指定
- ID 管理やバージョン管理が不要

### 2. Detect-Only モード

- セーフガード違反を検出するのみ
- ブロック、パス、再試行、ログ記録などのアクションをカスタマイズ可能
- 数値の重大度スコアと信頼度スコアを返却

### 3. 独立したセーフガード チェック

- プロンプト攻撃検出は、ジェイルブレイク、プロンプトインジェクション、プロンプトリークの 3 つの個別チェックとして実装
- 各チェックを独立して呼び出し可能

## 最適なユースケース

### 対象ユーザー

- エージェンティック AI アプリケーションの複数ステップで異なるリスク プロファイルに対応する必要があるユーザー
- ワークフロー進化に応じてチェックを柔軟に追加、削除、調整したいユーザー
- セーフガードを個別に独立して呼び出す必要があるユーザー
- コンテンツフィルタ、プロンプト攻撃検出、機密情報フィルタなどのセーフガードをカスタマイズして実装する必要があるユーザー

---

## メリット・効果

- **運用シンプル化**: ガードレール ID 管理とバージョン管理が不要
- **柔軟性向上**: リクエストごとにセーフガード構成を動的に変更可能
- **コスト効率**: リソース作成・管理のオーバーヘッド削減
- **拡張性**: ワークフロー進化に対応した段階的なセーフガード追加が容易
- **グラニュラ制御**: 個別のセーフガードを独立して制御可能

---

## まとめ

### InvokeGuardrailChecks API のポジショニング

- エージェンティック AI アプリケーション向けの新しいセーフガード実装方式
- リソースレス設計により運用負荷を軽減
- ステップごとのきめ細かいリスク管理を実現

### 次のステップ

1. 既存のエージェンティック AI ワークフローで適用箇所を検討
2. API ドキュメンテーション確認とテスト実施
3. パイロット プロジェクトでの検証
4. 本番環境への段階的デプロイ
