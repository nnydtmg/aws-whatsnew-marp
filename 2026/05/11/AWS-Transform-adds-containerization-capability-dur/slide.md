# AWS Transformが移行中のコンテナ化機能をサポート

AWSへの移行と近代化を並行して実行

**発表日: 2026年5月11日**

---

## 概要

### 新機能のポイント

- **ソースコード自動分析**: GitHub、Bitbucket、GitLab、ZIPファイルなどから自動的にコードを分析
- **Docker化の自動化**: Dockerfileを自動生成し、CVE脆弱性スキャン統合
- **クラウドネイティブ化**: Amazon ECR、Amazon ECS、Amazon EKSへのデプロイに対応
- **IaC生成**: Terraform IaCとHelmチャートをデプロイ準備完了の状態で生成
- **スケール対応**: 数千アプリケーションの大規模コンテナ化に対応

---

## 前提・背景

### 移行の課題

- **時間がかかる**: 手動のコンテナ化は複雑で時間を要する
- **リスク**: コード分析と依存関係管理の誤りが発生しやすい
- **複雑性**: モノレポ・マルチレポなど異なる構造への対応が必要

### 市場動向

- クラウドネイティブアーキテクチャへの需要が拡大
- リフト&シフト（リホスト）と近代化の並行実行が求められる
- AIエージェントによる自動化が、エンタープライズ領域で加速

---

## 主な機能

### 1. ソースコード分析と自動化

- リポジトリの自動分析で言語・フレームワーク・依存関係を検出
- 最適化されたDockerfileの自動生成
- CVE脆弱性スキャン統合でセキュリティも確保

### 2. 複数のリポジトリ構造に対応

- **モノレポジトリ**: 単一リポに複数プロジェクトがある場合も対応
- **マルチリポジトリ**: 複数リポを一括処理
- **プライベート依存関係**: AWS CodeArtifactで解決

---

## 主な機能（続き）

### 3. インフラストラクチャコード生成

- **Terraform**: デプロイ準備完了のIaCを生成
- **Helm チャート**: Kubernetes デプロイメント用の設定を自動生成
- 環境差分に対応した複数環境への展開が容易

### 4. 移行ウェーブ計画との統合

- 移行ウェーブ計画内でアプリケーションを以下のいずれかに割り当て可能
  - **リホスト**: VMMigration Service での従来の移行
  - **コンテナ化**: 新しいコンテナ化パスを選択

## ユースケース

### 1. レガシーアプリケーション移行

- オンプレミスの .NET、Java、Python アプリケーション
- 迅速にコンテナ化して Amazon EKS で運用

### 2. 大規模マイグレーション

- 数百～数千のアプリケーション
- 移行ウェーブに基づいた段階的な実行
- 一部リホスト、一部コンテナ化の並行実行

### 3. クラウドネイティブ化

- Amazon ECS/EKS への完全移行
- マイクロサービス化への第一歩

---

## 利用開始

### 利用可能なリージョン

- AWS Transform が提供されているすべての AWS リージョンで利用可能

### 次のステップ

1. AWS Transform にアクセス
2. ソースコードリポジトリを接続
3. コンテナ化対象アプリケーションを選択
4. 自動分析と Dockerfile 生成を実行
5. Amazon ECR、ECS、EKS へデプロイ

## まとめ

### AWS Transform コンテナ化の位置づけ

- **リホスト（リフト&シフト）と近代化の橋渡け**: 移行と近代化を並行実行
- **自動化による効率化**: AI エージェントが手作業を大幅削減
- **スケール対応**: 数千アプリケーションの一括処理に対応
- **セキュリティ確保**: CVE スキャン統合で本番運用レディ

### 今後の展望

- クラウドネイティブアーキテクチャへの加速
- エンタープライズ マイグレーション プログラムの効率化
- コスト削減と運用負荷軽減の実現
