# Amazon Bedrock AgentCore Runtime
持ち込みファイルシステム対応（S3 Files / EFS）

**発表日: 2026年5月6日**

---

## 概要

### 3つの主要な改善

- **Amazon S3 FilesとAmazon EFSの持ち込み対応**: 開発者が自身のファイルシステムをエージェントランタイムに直接アタッチ可能
- **シームレスなファイル操作**: カスタムコード不要で、標準的なPOSIX操作でファイルの読み書きが可能
- **複数セッション間でのデータ共有**: スキル、テンプレート、データセットをセッション・エージェント間で永続化・共有

---

## 前提・背景

### AI エージェント開発の課題

- **データの分散管理**: 複数のセッション・エージェント間でデータを共有することが困難
- **セッション終了時のデータ喪失**: エージェントが生成したコード、パッケージ、アーティファクトが失われる
- **複雑なストレージ連携**: オブジェクトストレージとファイルシステムのギャップを埋める複雑なパイプラインが必要

### 最近の関連動向

- **Amazon S3 Files（2026年4月）**: S3をフル機能で高性能なファイルシステムに
- **AgentCore Runtime 管理セッションストレージ（2026年3月プレビュー）**: 停止・再開サイクル全体で状態を永続化
- **エージェント間協業の需要増加**: 複数のエージェントが共同でタスクを実行する必要性

---

## 主な機能

### Amazon S3 Files マウント

- **標準ファイルシステムアクセス**: S3バケットを通常のLinuxファイルシステムのように操作
- **自動同期**: ファイル操作とS3 APIの両方でアクセス可能、変更は自動的に同期
- **高性能**: sub-millisecond レイテンシー、マルチTB/sの読み込みスループット

### Amazon EFS アクセスポイント マウント

- **専用共有NFS**: 複数エージェント・セッション間での共有に最適
- **POSIX準拠**: 標準的なファイルシステム操作が全てサポート
- **NFS一貫性**: close-to-open 一貫性をサポート

---

## 利用シーン

### セッション開始時のリソース読み込み

- **スキル（Skills）**: 共有スキルセットをセッション開始時に読み込み
- **プロンプトテンプレート**: キュレーションされたテンプレートの再利用
- **参照データセット**: 毎回のダウンロードが不要

### 複数セッション・エージェント間でのデータ共有

- **データセット協業**: 複数のエージェント/セッションが同一データセットで協業
- **ナレッジベース共有**: 組織全体のナレッジベースを共有リソースとして活用
- **ツールライブラリ統一**: 複数プロジェクト間でツールライブラリを統一管理

### 長時間ワークフロー

- **中間結果の永続化**: 長時間実行されるワークフローで進捗を保存
- **セッション再開**: 中断後、以前の状態から作業を継続
- **成果物の保護**: ビルドアーティファクト、gitヒストリーなども保護

---

## セットアップと設定

### 必要な要件

- **VPC設定**: AgentCore Runtime が VPC 内で設定されている
- **アクセスポイントARN**: S3 Files または EFS アクセスポイントのARN
- **IAM パーミッション**: 実行ロールに s3files:* や elasticfilesystem:* 権限

### マウント仕様

- **マウントパス**: 開発者が指定したパス（セッションごと）
- **セッション単位**: 各セッションが独立したマウントポイント
- **ファイルシステム操作**: open(), read(), write(), os.listdir(), shutil.copy() など標準POSIX操作

## パフォーマンス特性

### レイテンシーと一貫性

| 項目 | 仕様 |
|------|------|
| **レイテンシー** | sub-millisecond（アクティブデータ） |
| **一貫性モデル** | NFS close-to-open 一貫性 |
| **スループット** | マルチTB/s（S3 Files） |
| **並行接続** | 数千のコンピュートコネクション |

## メリット・効果

### 開発効率の向上

- ✅ **カスタムコード不要**: マウントコード、特権コンテナ不要
- ✅ **ダウンロードオーケストレーション廃止**: エージェント起動前のセットアップが不要
- ✅ **標準ツール対応**: git、pip、npm など既存ツールがそのまま動作

### 運用の簡素化

- ✅ **データの一元管理**: 複数セッション・エージェント間でのデータ共有が容易
- ✅ **ストレージの統合**: オブジェクト/ファイルストレージ間のギャップを解消
- ✅ **長期作業の保護**: セッション終了後も中間結果が保護される

### スケーラビリティ

- ✅ **多数並行接続**: 数千のエージェント/セッション並行実行
- ✅ **S3の拡張性**: S3バケットのスケーラビリティをフルに活用
- ✅ **共有リソース最適化**: ストレージ重複が排除される

## ユースケース例

### 1. エンタープライズ AI アシスタント

- **組織ナレッジベース**: エンタープライズナレッジベースを複数のエージェント間で共有
- **スキルセット**: 部門別・プロジェクト別のスキルを集中管理
- **セッション永続性**: 重要な作業履歴を自動保護

### 2. RAG（検索強化生成）パイプライン

- **データレイク直接アクセス**: S3 Files でデータレイクに直接アクセス（ステージング不要）
- **複数ソース統合**: 複数の RAG エージェントが同一データセットを活用
- **低レイテンシー**: sub-millisecond アクセスで高速検索

### 3. マルチエージェント協業

- **タスク分割**: 複数エージェントが連携して複雑なタスクを実行
- **中間成果物共有**: エージェント A の出力をエージェント B が入力として活用
- **進捗トラッキング**: 共有ファイルシステムで全進捗を管理

## まとめ

### Amazon Bedrock AgentCore Runtime のパラダイムシフト

- **ファイルシステムの自由度向上**: S3 Files と EFS の両オプション
- **セッション管理の進化**: 管理セッションストレージと持ち込みファイルシステムの選択肢
- **エンタープライズ対応**: 複数エージェント・セッション間でのシームレスなデータ共有

### 次のステップ

1. **既存エージェント評価**: 現在のエージェント設計で活用可能な点を検討
2. **パイロット実装**: S3 Files または EFS を使用したテストプロジェクト
3. **スキル・データ整理**: 共有可能なスキル、テンプレート、ナレッジベースを整理
4. **本番展開**: 複数セッション・エージェント間での協業パターン実装
