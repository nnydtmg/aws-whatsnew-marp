# Amazon Quick Automate

APIによる自動化ジョブのトリガーと監視が実現

**発表日: 2026年4月21日**

---

## 概要

### 新しく提供されるAPI機能

- **StartAutomationJob API**: 外部アプリケーションから自動化ジョブをプログラムで起動
- **DescribeAutomationJob API**: 実行中の自動化ジョブのステータスを監視・確認
- スケジュール実行を超えた、イベント駆動型の自動化が可能に

### 主なメリット

- 既存システムやイベント駆動型アーキテクチャへのシームレスな統合
- データパイプラインやワークフロー調整の効率化
- バッチ操作の一元管理

---

## 前提・背景

### 従来の課題

- Amazon Quick Automateはスケジュール実行のみに対応
- 外部アプリケーションからのトリガーが不可能
- 複数のシステムにまたがるワークフロー統合が困難

### 市場の需要

- Agentic AIとイベント駆動型ワークフローへの需要拡大
- 自動化システムとの統合ニーズの増加
- 複雑な業務プロセスの自動化要件

---

## 主な機能と改善

### StartAutomationJob API

- **目的**: 外部アプリケーションから自動化ジョブを起動
- **機能**:
  - カスタム入力データでのジョブ呼び出し
  - 動的入力パラメータのサポート（型付きスキーマ）
  - 構造化された結果の取得

### DescribeAutomationJob API

- **目的**: ジョブの実行状況をリアルタイムで確認
- **機能**:
  - ジョブステータスの監視
  - 完了時の結果取得
  - 進捗状況の追跡

## 利用パターンと活用例

### パターン1: イベント駆動型トリガー

- 新規ユーザー登録イベント → 自動化ジョブ起動
- 注文完了イベント → 処理自動化
- システムアラート → インシデント対応自動化

### パターン2: データパイプライン統合

- ETLワークフローの一部として自動化を組み込み
- 複数ステップの処理を自動実行
- 結果をダウンストリームシステムに連携

### パターン3: バッチ処理と多重パラメータ

- 複数の異なる入力パラメータでバッチ実行
- 単一アプリケーションから大量実行管理
- 効率的なリソース活用

### パターン4: マルチサービスのワークフロー調整

- AWS サービス間のワークフロー統合
- サードパーティアプリケーションとの連携
- エンドツーエンドのプロセス自動化

## 主なユースケース

### UC1: SaaS 管理画面

- ユーザーの申請ボタン押下 → Quick AutomateへのAPI呼び出し
- 申請内容の自動処理 → 結果をユーザーに表示

### UC2: データパイプライン

- スケジューラー → 複数の自動化ジョブを連鎖実行
- 各ジョブの結果を次ステップに受け渡し
- エラーハンドリングと再試行の自動化

### UC3: イベント駆動型ワークフロー

- EventBridge / SNS → Quick Automate API 連携
- イベント発生時に即座に自動化を実行
- 複数アプリケーション間のプロセス調整

## 技術仕様

### API呼び出しの基本フロー

```
外部アプリケーション
    ↓
StartAutomationJob API (入力パラメータ指定)
    ↓
Quick Automate (ジョブ実行)
    ↓
DescribeAutomationJob API (ステータス確認)
    ↓
結果取得・後続処理
```

### パラメータの型付けスキーマ

- 入力: 型定義されたスキーマに基づくパラメータ
- 出力: 構造化されたJSON形式の結果
- バリデーション: スキーマに基づく自動検証
