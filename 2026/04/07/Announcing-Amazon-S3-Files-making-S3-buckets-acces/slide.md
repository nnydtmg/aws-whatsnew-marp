# Amazon S3 Files
S3バケットをファイルシステムとしてアクセス可能に

**発表日: 2026年4月7日**

---

## 概要

### Amazon S3 Filesの3つの主要特徴

- **ファイルシステムアクセス**: S3バケット内のデータに直接ファイルシステムとしてアクセス可能
- **シームレス統合**: ファイルシステムAPIとS3 APIの両方で同時にアクセス可能
- **高パフォーマンス**: 複数テラバイト/秒の集約読み取りスループット

### 対応リージョン

- **34のAWSリージョンで一般提供開始**

---

## 前提・背景

### 市場の動向

- S3は20年の歴史を持ち、500兆以上のオブジェクトを保存中
- AI/ML分野でのデータアクセス方法の多様化が進行
- ファイルベースのアプリケーションとオブジェクトストレージ間の連携需要が増加

### 従来の課題

- ファイルシステムベースのアプリケーションがS3データにアクセスする際、複製やステージングが必要
- 複数のコンピュートリソース間でのデータ共有が複雑
- オブジェクトストレージとファイルシステム間のデータ同期の複雑性

---

## 主な機能と特徴

### 1. ファイルシステムセマンティクス

- AWS計算リソースからS3バケット内のデータに直接ファイルシステムとしてアクセス
- ファイルシステムの完全なセマンティクスと低レイテンシーパフォーマンスを実現
- 標準的なファイルシステムAPIを使用可能

### 2. 効率的なデータ操作

- **データ複製不要**: S3に保存されたままでアクセス
- **ステージング削除**: 直接アクセスにより準備ワークロード効率化
- **高スループット**: 複数テラバイト/秒の集約読み取り対応

### 3. インフラ構成

- **Amazon EFS採用**: スケーラビリティと費用対効果を備えた設計
- **デュアルAPI対応**: ファイルシステムAPIとS3 API両立

## このアップデートが適している対象

### 利用を推奨するシーン

- **ファイルベースアプリケーション**: ファイルシステムAPIを使用する既存ツールやアプリケーション
- **データレイク環境**: S3にデータレイクと分析データを保存している企業
- **AIエージェント**: メモリを永続化し、パイプライン間で状態を共有する必要がある場合
- **MLチーム**: ファイルの複製やステージングなしにデータ準備ワークロード実行
- **共有アクセス**: 複数の計算リソースから同じS3データへの共有アクセスが必要な場合
- **複雑性削減**: ファイルシステムとオブジェクトストレージ間のデータ同期の複雑性排除

---

## メリット・効果

### パフォーマンス面

- **低レイテンシー**: ファイルシステム並みのレスポンス時間
- **高スループット**: 複数テラバイト/秒の集約読み取り対応
- **直接アクセス**: データ複製/ステージングによるオーバーヘッド削減

### 運用面

- **複雑性削減**: S3とファイルシステム間のデータ同期が不要
- **資本効率**: 複製ストレージの不要化によるコスト削減
- **スケーラビリティ**: EFSベースによる無制限なスケーリング対応

### アプリケーション面

- **既存ツール利用**: ファイルシステムベースのツール・アプリケーションの直接利用可能
- **マルチテナント**: 複数リソースからの同時アクセス対応
- **共有状態管理**: AIエージェント/MLパイプライン間の状態共有が容易

## ユースケース

### 1. データレイク分析

```
S3 Data Lake → S3 Files → Analytics Tools
              (複製不要)
```

複数のECインスタンスやSageMakerノートブックから、S3データレイクにファイルシステムとしてアクセス。複製やステージングが不要になり、分析処理が迅速化。

### 2. AIエージェント・メモリ永続化

```
AI Agent → S3 Files → S3 Storage
    (メモリ永続化・共有)
```

AIエージェントが状態や中間結果をS3に永続化。複数エージェント間での状態共有が容易に。

### 3. MLパイプライン・データ準備

SageMaker Processing、SageMaker Training、Lambdaなど複数のコンピュートリソースが同一S3データセットにアクセス。データ同期の複雑性が大幅に削減。
