# Amazon CloudFront
SHA-256署名URLとクッキーサポート

**発表日: 2026年4月1日**

---

## 概要

### CloudFrontの署名付きコンテンツ配信が強化

- **SHA-256ハッシュアルゴリズムのサポート**: 署名付きURLと署名付きクッキーで利用可能
- **セキュリティ向上**: SHA-1よりも強力な衝突検出機能を提供
- **完全な下位互換性**: 既存のSHA-1ベースの署名付きURLとクッキーは継続稼働
- **追加コストなし**: SHA-256署名の利用に料金は発生しません
- **グローバル対応**: すべてのCloudFrontエッジロケーションで利用可能

---

## 前提・背景

### セキュリティコンプライアンス要件の高まり

- デジタル署名にSHA-256を必須とする規制やコンプライアンス要件が増加
- 組織のセキュリティポリシーがSHA-1から移行を要求
- SHA-1の暗号化上の弱点が既知問題として指摘されている

### 市場の動向

- **2025年9月**: CloudFrontはECDSAサポートを発表（RSA/ECDSAの選択肢)
- **2026年4月**: SHA-256ハッシュアルゴリズムをサポート
- エンタープライズ環境での暗号化標準の統一への動き

---

## 主な機能・変更内容

### SHA-256署名の使用方法

#### 署名付きURLでの指定
```
Hash-Algorithm=SHA256
```
クエリパラメータをURLに追加して指定

#### 署名付きクッキーでの指定
```
CloudFront-Hash-Algorithm=SHA256
```
クッキー属性として指定

### 従来との互換性

| 項目 | SHA-1 (従来) | SHA-256 (新) |
|-----|-----------|----------|
| ハッシュアルゴリズム | 160ビット | 256ビット |
| 衝突検出 | 弱い | 強力 |
| コンプライアンス | 限定的 | 最新標準に準拠 |
| 下位互換性 | - | 完全に保証 |

## 効果・メリット

### セキュリティ面でのメリット

- **強化されたセキュリティ**: より長いハッシュ値により衝突の可能性が極めて低い
- **規制対応**: 業界標準やコンプライアンス要件への準拠
- **将来対応**: 暗号化標準の進化に対応

### 運用面でのメリット

- **段階的な移行**: 既存コンテンツは自動的にSHA-1で動作継続
- **柔軟な選択肢**: SHA-256の必要な箇所だけ選択的に適用
- **コスト効率**: 追加コストなしで利用可能

### ビジネス面でのメリット

- **顧客信頼の向上**: より安全なコンテンツ配信
- **エンタープライズ対応**: 厳格なセキュリティポリシーへの対応
- **リスク低減**: サイバーセキュリティ脅威からの保護

## 実装方法（クイックガイド）

### Step 1: CloudFront Key Pair生成

```
SHA-256対応のキーペア生成
```

### Step 2: 署名付きURLの生成

```python
# Python例
url = generate_signed_url(
    key=your_private_key,
    algorithm='SHA256',  # SHA256を指定
    expires_at=datetime.now() + timedelta(hours=1)
)
```

### Step 3: クッキーの設定

```
CloudFront-Hash-Algorithm: SHA256
CloudFront-Policy: [policy]
CloudFront-Signature: [signature]
```

## まとめ

### Amazon CloudFront SHA-256署名サポートの意義

- **セキュリティ強化**: 現代的な暗号化標準への対応
- **完全互換**: 段階的な移行が可能な設計
- **エンタープライズ対応**: コンプライアンス要件への準拠

2. **テスト環境での検証**: 新規署名生成とクライアント側での動作確認
3. **段階的ロールアウト**: 既存環境への影響を最小化した導入
