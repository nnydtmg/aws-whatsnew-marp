# Amazon Bedrock バッチ推論
Converse API フォーマット対応

**発表日: 2026年2月27日**

---

## 概要

### Amazon Bedrock バッチ推論の進化

- **Converse API フォーマット対応**: バッチ推論が Converse API フォーマットをサポート開始
- **統一されたリクエストフォーマット**: リアルタイム推論とバッチ推論で同じフォーマットを使用可能
- **プロンプト管理の簡素化**: 複数モデル間での切り替え労力が大幅に削減
- **既存モデルの拡張**: 標準的な Converse API リクエストフォーマットで構成可能

---

## 前提・背景

### これまでの課題

- バッチ推論では **InvokeModel API** を使用した**モデル固有のリクエストフォーマット**が必須
- リアルタイム推論と異なるフォーマットが必要で、**開発効率が低下**
- 複数のモデルを使用する場合、**切り替えに大きな労力**が必要
- **プロンプト管理が複雑化**し、保守コストが増加

### 市場の動向

- エンタープライズにおけるバッチ処理ワークロードの増加
- AI/ML パイプラインの標準化への要望
- 複数モデル間の相互運用性に対する需要拡大

---

## 主な機能

### Converse API フォーマット対応

- **統一されたリクエストフォーマット**: リアルタイム推論とバッチ推論で同じフォーマットを使用
- **標準的な API リクエスト構造**: `messages`、`systemPrompt`、`inferenceConfig` など
- **Converse API レスポンスフォーマット**: 出力も Converse API 形式に統一
- **モデルに依存しないフォーマット**: Claude、Llama など複数モデルに対応

### 実装方法

- **Amazon Bedrock コンソール**: バッチ推論ジョブ作成時に **Converse** をモデル呼び出しタイプとして選択
- **API での設定**: バッチ推論 API で `modelInvocationType` を **Converse** に指定
- **既存バッチジョブとの互換性**: InvokeModel 形式との共存が可能

---

## パフォーマンス・対応状況

### 利用可能なリージョン

- **対応範囲**: Amazon Bedrock バッチ推論をサポートする**すべての AWS リージョン**で利用可能
- **段階的なロールアウト**: 既存バッチ推論ユーザーに自動提供

### 対応モデル

- Anthropic Claude (全バージョン)
- Open-weight モデル (Llama など)
- その他 Converse API 対応モデル

---

## 利用方法（使用例）

### コンソール での設定

1. Amazon Bedrock コンソール → バッチ推論
2. 「バッチ推論ジョブを作成」をクリック
3. モデル呼び出しタイプで **Converse** を選択
4. 標準的な Converse API リクエストフォーマットで入力データを指定

### リクエストフォーマット例

```json
{
  "messages": [
    {
      "role": "user",
      "content": "このテキストを分類してください"
    }
  ],
  "inferenceConfig": {
    "maxTokens": 512,
    "temperature": 0.7
  }
}
```

## 効果・メリット

### 開発効率の向上

- **プロンプト管理の簡素化**: 同じプロンプトをリアルタイムとバッチで再利用可能
- **開発コストの削減**: フォーマット変換処理が不要に
- **学習曲線の短縮**: 開発者が統一された API を学習すればよい

### 運用効率の向上

- **モデル切り替えが容易**: 別のモデルへの乗り換え時に形式変更が不要
- **エラーハンドリングの統一**: リアルタイムとバッチで同じロジックを使用
- **テストコストの削減**: 単一のテストスイートで対応可能

### スケーラビリティ

- **大規模バッチ処理の簡素化**: 複数モデル間での一括処理が可能
- **複数チーム間の協業**: 統一されたフォーマットで調整が容易
